# 项目文档约束

## 快速上下文

- 这是一个 VitePress 技术文档项目，主要内容在 `docs/`，站点导航配置在 `.vitepress/navbar.ts` 和 `.vitepress/sidebar.ts`。
- 静态资源放在项目根目录 `public/` 下（不是 `docs/public/`），按主题建子目录（如 `public/protocol/`、`public/design-pattern/`）。Markdown 中使用站点绝对路径引用，例如 `/protocol/osi-tcpip-layers.svg`。
- 新增专题时通常需要同时处理三件事：新增 Markdown 内容、必要时补图示资源、补 VitePress 侧边栏或顶部导航入口。
- 常用验证命令是 `npm run docs:build`。如果构建失败，要说明失败点是否由本次改动引入。

## 导航结构规范

- 顶部导航（navbar）每个分类只放一个入口链接，不在下拉里展开所有子页面。用户点击后通过左侧边栏（sidebar）浏览子分类和子页面。
- 同一个主题的所有子路径共享一棵侧边栏树（用公共前缀匹配，如 `/docs/devlanguage/java/`），不按子目录拆成多个独立 sidebar 键。
- 侧边栏二级分类使用 `collapsed: true` 默认折叠，一级分类 `collapsed: false` 默认展开。

## 写作偏好

- 所有新增和修改的正文内容默认使用中文记录，除技术名词、命令、代码、API、链接、包名、配置项等必须保留原文的内容外，不主动混入英文叙述。
- 文档内容应保持技术知识库口吻，不出现“按某某要求”“根据用户要求”“本文由 AI 生成”“下面为你整理”等生成过程、任务来源或对话痕迹。
- 不使用“一句话总结”“一句话理解”“一张图看清”“经验法则”“让我们”“显而易见”“众所周知”“值得一提”等 AI 味重的套话。章节标题直接用“总结”“对比”“选型”等朴素词汇。
- 这是技术知识库，不是面试网站。不写“面试高频题”“八股文”“常见考点”等内容。能力评估类内容应改写为“达成标志/能力验证清单”，从“能独立完成什么”角度描述。
- 科普类文章可以使用通俗解释，但应以官方资料、主流技术事实和工程实践为依据；涉及版本、发布时间、规范变化等易变信息时，需要核对权威来源并写清具体日期或版本。
- 新增专题应优先保持现有目录结构和 VitePress 导航风格，入口页负责总览导读，细节页按主题拆分。
- 文档写作要避免空泛总述堆叠。每个专题尽量沉淀为“概念边界 -> 操作入口/配置位置 -> 可复制命令或配置 -> 典型场景 -> 排错闭环”的结构。
- 用户面向的正文不写资料搜集过程、参考来源列表、校准来源、依赖某资料等说明；来源核对只作为写作过程，必要时把稳定事实转成正文结论。
- 涉及容易混淆的流程、层级、生命周期、数据流或排查路径时，在 `public/<主题>/` 下补充 SVG 图示，并在正文中用绝对路径引用，例如 `/protocol/tcp-vs-udp.svg`。
- SVG 图示应服务理解，不做装饰；优先表达决策链、状态流转、组件关系、命令作用范围。

## 内容落地方式

- 对“科普、整体介绍、体系化整理”类需求，不只列大纲；应补充可操作的判断标准、代码前后对比、配置示例、排查步骤或选型路线。
- 对“优化润色”类需求，优先提升可读性、结构完整性和实操价值；不要只替换措辞。
- 对外发布的科普文章要保持专业、克制的标题风格，避免“高效理解”“快速搞懂”“一文看懂”等口语化标题。
- 正文不要太泛，不要堆砌过多标题和列表；优先用连续段落解释背景、取舍、适用边界和落地步骤，列表只用于对比、检查清单、步骤和关键信息压缩。
- 入口页要承担导读职责：说明这个专题解决什么问题、如何阅读、各子页面分别看什么。
- 细节页要尽量给出真实工程场景，例如订单、支付、导出、通知、配置、部署、排错、性能优化等。
- 对容易误用的技术点，要写清适用场景、不适用场景、常见误区和判断标准。

## 文档改造沉淀

- 优化已有模块时，先看目录、现有页面、侧边栏和相关命令页，明确“这个模块负责讲机制，哪些内容应链接到已有详细页”，避免重复堆命令或把概述写成杂糅清单。
- 入口页应承担主题地图职责：说明模块解决什么问题、核心概念之间的关系、阅读路径和细节页入口。没有覆盖到的子模块，应在概述中补到详细说明的链接。
- 细节页优先用“对象关系、生命周期、数据流、排查路径”解释技术。代码和命令只保留能定位入口、验证现象、完成闭环的少量片段。
- 图示只在能降低理解成本时添加，优先画组件关系、状态流转、请求链路、命令作用范围和排查分支。图示文件放在 `public/<主题>/`，正文使用站点绝对路径引用。
- 对同一主题的多篇文章，应先分清职责：概述讲全局模型，机制页讲边界和运行过程，实操页讲配置和排障，命令页讲可复制用法。
- 改造完成后检查三类问题：是否有空泛标题堆叠、是否有生成式口吻或对话痕迹、是否存在失效链接。最后执行 `npm run docs:build`，并区分本次改动问题和工作区已有问题。
- 当用户指出一种更好的写作、检查或协作方式时，应默认沉淀到本文件；后续交互中发现能稳定提升质量的做法，也应在不影响当前任务的前提下及时补充到本文件。
- 重构已有内容时，优先形成“科普总览 + 模型详细说明”的阅读结构：总览页讲问题背景、边界、阅读路径和整体图；模型页用一条真实业务线贯穿概念、判断标准、代码位置和协作方式。
- 面向外部发布的科普内容要少写概率式、猜测式表达，多用具体场景、业务事实、状态流转、对象关系和图示解释。能用“订单已支付后库存如何响应”说明的问题，不写成抽象口号。
- 对历史页面做较大改造时，保留旧入口的可达性，但应把正式入口和侧边栏调整到更清晰的新模块；旧路径可以改成简短说明或迁移页，避免读者进入过时内容。
- 图示完成后要检查尺寸比例和引用路径。架构、流程、模型类 SVG 优先使用横向阅读比例，避免过窄、过高或元素拥挤。
- 如果站点构建失败但失败点来自工作区已有死链，应明确说明失败文件和链接；必要时再做本次新增页面的资源、链接和关键词扫描，证明本次改动范围内没有新增问题。

## 协作方式

- 默认直接修改项目文件并验证，不停留在建议层面；除非需求明显是在讨论方案。
- 改动前先快速查看现有目录、风格和导航配置，保持与已有文档体系一致。
- 工作区可能已有其他未提交改动，不回退无关文件，只处理当前任务相关文件。
- 最终回复简洁说明改了哪些内容、关键文件路径、验证命令和结果。
