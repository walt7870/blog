# Claude Code 推荐实践：从三个开源 Skill 看清“怎么把能力拉满”

这篇不是“再推荐一批 Skill 清单”，而是把一篇实战文章里的核心内容拆开讲清楚：  
为什么有些 Skill 一看就“工程味很重”，以及这些做法怎么迁移到你自己的 Claude Code 工作流里。

文章主角是一个开源集合 `garden-skills`，其中三个 Skill 分别对应网页生成、本地知识库检索和 AI 生图。它的价值不在于“功能多”，而在于把大量踩坑经验写进了 `SKILL.md` 规则里。

## 先说结论：强 Skill 的分水岭是“约束是否具体”

普通 Skill 往往写“建议怎么做”。  
这套 Skill 的特点是写“禁止怎么做”，而且禁令背后有明确的运行时原因。比如作用域冲突、iframe 滚动异常、token 浪费等。  
这就是它比“提示词锦集”更实用的地方。

## 一、`web-design-engineer`：不是让 AI 会写页面，而是写得不“AI 味”

这个 Skill 的目标很直接：避免“看一眼就知道是 AI 生成的网页”。

它做了四件很关键的事：

- **反套路黑名单**：直接禁掉高频 AI 视觉套路（紫粉渐变、统一字体风格、emoji 充图标等）
- **先声明设计系统**：必须先给配色、字体、间距、圆角、阴影，再让用户确认
- **v0 原型优先**：先快速搭骨架对齐方向，不追求一次出“终版”
- **高级模式库沉淀**：把可复用的动画、交互、布局模式放进 references

更值得学的是它的“硬约束”：

- 在多脚本场景禁用易冲突变量命名，避免作用域污染
- 禁用在 iframe 中容易破坏外层滚动的调用方式

这些规则不是“风格偏好”，而是稳定性策略。

## 二、`rag-skill`：本地检索不是“把文件全塞进上下文”

这个 Skill 解决的是一个常见问题：项目里有 `knowledge/`，文档很多（尤其 PDF / Excel），但上下文窗口有限。

它的设计核心可以概括成三条：

- **先索引后深入**：先读目录级结构文件，再决定钻哪个子目录
- **先学习再处理**：碰到 PDF / Excel，先读处理规范文档，再执行提取
- **渐进式检索**：小步 grep + 局部读取，最多限定轮数，避免无限扩展

最有启发的一点是“工具边界意识”：

- 目录是否存在，用目录检测命令，不用文件匹配结果去猜

这类规则看起来细，但能直接减少误判和 token 浪费。

## 三、`gpt-image-2`：同一个 Skill，按环境自动降级

这个 Skill 的亮点不是“能生图”，而是“知道自己在哪种运行环境里”。

它按环境分三种模式：

- **模式 A**：本地环境完整，可端到端执行出图
- **模式 B**：本地不完整，但宿主有图像工具，Skill 退化为提示词和参数层
- **模式 C**：都没有，退化为高质量 prompt 输出

本质上，这是在做“能力协商”：  
有能力时全流程执行，能力不足时优雅降级，而不是直接失败。

这个思路对很多 Skill 都适用，不只限于生图。

## 四、这三个 Skill 共同体现的工程方法

把三个案例放在一起看，会发现它们共享一套方法：

- **先定义边界，再执行任务**
- **把经验写成硬规则，而不是口头建议**
- **每个环节都考虑 token 成本**
- **允许降级路径，避免一条链路全盘失败**
- **把可复用模式沉淀到 references，而不是重复发明**

如果你在写自己的 Skill，这五条几乎可以直接当检查清单。

## 五、怎么在 Claude Code 里借鉴这套做法

不一定要照搬仓库，你可以先做一个最小版本：

1. 给你的 Skill 加“禁止行为”段落（写具体，不写空话）  
2. 把高频输入输出格式固定成模板  
3. 为“环境不满足”设计降级路径  
4. 把踩过的坑沉淀到 references，强制先读再执行

执行一段时间后，你会明显感觉：  
Claude Code 不再像“随机发挥”，而是越来越像“按规程协作”。

## 六、适合先试哪一个

- 你经常让 AI 生成页面：先试 `web-design-engineer`
- 你有大量本地文档要检索：先试 `rag-skill`
- 你要做稳定的 AI 生图流程：先试 `gpt-image-2`

别贪多，一次跑通一个场景，效果会更直观。

## 结语

这篇案例真正有价值的，不是“又多了三个 Skill”，而是它展示了一种写 Skill 的方式：  
**把经验做成规则，把规则做成可执行流程。**

当 Skill 从“提示词集合”变成“工程规程”时，Claude Code 的上限才会被真正拉高。
