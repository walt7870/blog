# 组织流程工程：让规格成为团队常识

## 定义

组织流程工程是把 AI 编程从个人工具升级到团队系统的实践。

把个人的好习惯、决策标准、风险控制沉淀成团队的规则和流程。

## 为什么需要组织流程工程

### 个人 vs 团队的差异

```
个人开发者用 AI：
"我知道这个项目的规则，AI 也知道了，我们配合很好。"

团队用 AI：
"新来的工程师怎么知道规则？
 实习生给 AI 的提示词靠谱吗？
 A 同学和 B 同学让 AI 改同一个文件，结果冲突了怎么办？
 哪些改动可以让 AI 自动做，哪些必须人工审核？"
```

个人的隐性知识不能扩展到团队。需要把它们**显式化、文档化、工具化**。

## 组织流程工程的五个层面

### 1. 规格库和规范化

把好的规格保存和复用，建立规范。

#### 规格库结构

```
docs/
  specifications/
    api/
      user-auth.spec.md      # 用户认证 API 规格
      order-export.spec.md   # 订单导出规格
    features/
      payment-integration.spec.md   # 支付集成规格
    refactoring/
      api-layer.spec.md      # API 层重构规格
```

每个规格都包含：目标、范围、验收标准、风险点。

#### 规格评审流程

```
开发者：
1. 编写规格
2. 提交 review（设计文档 PR）

架构师 / 技术 lead：
3. Review 规格
   - 是否清晰可执行？
   - 是否有遗漏的风险？
   - 是否符合团队标准？
   
4. 通过后，规格作为后续 AI 编程的输入

基层开发者（含 AI）：
5. 按规格执行
```

### 2. 验收标准库

把重复出现的验收标准总结成 checklist 模板。

#### 常见的验收标准

```
通用：
☐ 代码通过类型检查
☐ 所有测试通过
☐ 没有 console 错误和警告
☐ 代码通过风格检查（Lint）

API 端点：
☐ 返回正确的 HTTP 状态码
☐ 返回数据格式符合规格
☐ 错误情况有合适的错误消息
☐ 有速率限制防止滥用
☐ 权限检查正确

数据库操作：
☐ 数据一致性得到保障
☐ 没有 N+1 查询
☐ 大数据量查询有索引优化
☐ 有数据备份和恢复能力

支付/权限相关：
☐ 人工复审通过
☐ 审计日志完整
☐ 有回滚方案
☐ 灾难恢复测试通过

UI 改动：
☐ 桌面和手机都能看
☐ 适配浅色和深色主题
☐ 无障碍检查通过
☐ 加载状态和错误状态都有提示
```

#### Checklist 模板

```markdown
## 验收 Checklist - [功能名称]

### 功能完成度
- [ ] 核心功能实现
- [ ] 边界情况处理
- [ ] 错误提示清晰

### 代码质量
- [ ] TypeScript 类型检查通过
- [ ] Lint 通过
- [ ] 测试覆盖率 >= 80%

### 性能和安全
- [ ] 不存在已知漏洞
- [ ] 没有性能退化
- [ ] 敏感操作有审计日志

### 文档
- [ ] 代码注释清晰
- [ ] 有实现文档
- [ ] 有运维说明
```

AI 或开发者改完后，用这个 Checklist 逐项验证。

### 3. 风险库和决策文档

把每个高风险决策和对应的控制措施记录下来。

#### 例：支付相关代码的风险控制

```
## 支付相关的风险和控制

### 风险 1：支付数据泄露
后果：违反 PCI DSS，法律问题，用户信任丧失

控制措施：
✓ 支付数据不能存储在我们的数据库（用 Stripe 托管）
✓ 涉及支付的所有改动需要两个高级工程师 review
✓ 支付代码不能由 AI 自动改，必须人工编写
✓ 生产环境的支付改动需要安全团队 sign-off

验证：
每季度的安全审计会检查支付数据的安全性。

### 风险 2：重复扣款
后果：用户损失金钱，投诉和退款

控制措施：
✓ 所有支付操作都有幂等性保证（相同请求不会重复扣款）
✓ 支付失败时有重试逻辑，但不会重复处理
✓ 数据库有 unique constraint 防止重复记录
✓ 有监控告警检测异常扣款

验证：
集成测试覆盖支付失败和重试场景。

### 风险 3：数据不一致
后果：订单记录和支付记录不匹配，无法对账

控制措施：
✓ 使用数据库事务保证原子性
✓ 支付和订单状态同步有超时处理
✓ 每日对账 job 检测不一致并告警
✓ 发现不一致时人工介入

验证：
每周对账报告，有不一致时会告警。
```

这份文档让新人和 AI 都知道为什么支付改动这么严格。

### 4. Skill 和自动化规则

把重复的工程任务自动化，写成 AI Skill。

#### Skill 例子

```
Skill: 新增 API 端点

目的：快速为 API 添加新端点，遵循团队规范

自动化内容：
1. 创建路由处理器（遵循命名规范）
2. 添加请求参数验证
3. 添加错误处理
4. 生成单元测试模板（至少 3 个场景）
5. 生成 API 文档

示例输入：
"新增 POST /api/orders/{id}/export 端点，
接受 format 参数（csv 或 excel），
返回下载链接。"

Skill 自动生成：
✓ src/api/handlers/orderExport.ts
✓ src/api/routes.ts（新增路由）
✓ test/api/orderExport.test.ts
✓ docs/api/export.md
✓ 错误处理和验证逻辑

然后开发者只需要填充业务逻辑。
```

#### Skill 库

```
skills/
  new-api-endpoint/          # 新增 API
  new-react-component/       # 新增 React 组件
  write-unit-tests/          # 补充单元测试
  optimize-query/            # 优化数据库查询
  migrate-old-pattern/       # 迁移过时代码到新模式
  security-audit/            # 安全审计（自动检查）
```

### 5. 用户反馈循环和持续改进

监控 AI 改动的效果，持续优化流程。

#### 关键指标

```
# AI 改动相关
✓ AI 改动通过率（多少百分比的改动一次通过 CI）
✓ AI 改动被 revert 的比例（多少百分比被后来回滚了）
✓ AI 改动的平均 review 时间（人工需要多久审核）
✓ AI 改动的平均质量评分（Reviewer 的满意度）

# 流程相关
✓ 规格的完整性评分（规格有遗漏吗）
✓ 验收标准的覆盖率（通过测试就一定能上线吗）
✓ Skill 的使用率（有多少百分比的任务用 Skill 完成）
✓ 重复 review 意见（多少意见在多个 PR 中重复出现）

# 业务相关
✓ 生产 bug 率（AI 改动导致的生产问题）
✓ 用户投诉（AI 改动相关的投诉）
✓ 性能指标（AI 改动是否影响了性能）
```

#### 改进循环

```
监控数据 → 发现问题 → 分析原因 → 改进流程 → 试运行 → 评估效果

例：
监控：发现 AI 改动有 20% 被 revert
分析：大多数是因为边界情况没处理
改进：在规格模板里加"边界情况处理"的 section
试运行：新的规格模板应用到下一批任务
评估：revert 率下降到 8%，改进有效

新知识：
把这个最佳实践加入文档和 Skill。
```

## 组织流程工程的三个层级

### L1：小团队（5-10 人）

必须有的：

```
✓ docs/CLAUDE.md or .claude/instructions.md
  （项目快速入门，禁区，核心命令）

✓ 基础的验收 Checklist
  （2-3 个常见场景）

✓ GitHub Issues / Linear for tracking
  （规格和任务有地方记）

✓ 基础 CI/CD
  （npm run test / build）

不需要有的：
（过度工程）
```

### L2：中等团队（10-50 人）

需要补充：

```
✓ docs/specifications/
  （关键功能的规格库）

✓ docs/ARCHITECTURE.md
  （系统设计和约束）

✓ docs/CONSTRAINTS.md
  （技术栈限制、依赖版本等）

✓ 完整的验收 Checklist
  （API、数据库、UI、安全等）

✓ 风险库（关键模块的风险和控制措施）

✓ 企业级 CI/CD
  （多环境，自动部署）

✓ 1-2 个高频 Skill
  （新增 API 端点、新增 React 组件）

✓ 团队规范文档
  （命名规范、代码风格、 Git 流程）
```

### L3：大型组织（50+ 人）

完整的生态：

```
✓ 完整的规格库和审核流程

✓ 完整的风险库和审计系统

✓ 多个 Skill 库（20+）

✓ 自动化评估系统
  （代码质量、性能、安全自动扫描）

✓ 完整的知识管理系统
  （Wiki、最佳实践、决策文档）

✓ AI 编程评估体系
  （跟踪效果，持续优化）

✓ 跨团队协调机制
  （共享库、依赖管理、大改动的 review）
```

## Spec-Driven 在组织中的落地

### 1. 规格作为事实来源

```
不要有这样的情况：
❌ 规格说一套，代码做另一套
❌ Jira 里的需求和规格不一致
❌ 会议纪要里有新的要求，但规格没改

要做的：
✓ 规格是真实的需求，其他地方都来自规格
✓ 规格改动需要 review 和通知
✓ 代码必须满足规格，不满足的是 bug
✓ 规格库定期清理（去掉已完成的、过时的）
```

### 2. 规格的版本管理

```
规格也是代码，放在 git 版本控制中：

docs/specifications/
  ├── v1/
  │   ├── payment-integration.md (deprecated)
  │   └── ...
  ├── v2/
  │   ├── payment-integration.md (current)
  │   └── ...
```

这样规格改动有历史记录，也能追踪为什么改了。

### 3. 规格评审的权限模型

```
规格改动需要谁 review：

关键性规格（支付、权限、架构）：
  ├─ 原作者
  ├─ 技术 lead
  ├─ 安全/合规团队
  └─ 产品经理

普通功能规格：
  ├─ 原作者
  ├─ 一个同行 review

Skill 和模板：
  └─ 原作者和至少一个高级工程师
```

## 组织 AI 编程的常见陷阱

### ❌ 流程过度复杂

```
"让 AI 改代码需要 15 个步骤和 8 个人 sign-off"

后果：
- 开发变慢，没有发挥 AI 的优势
- 人工 review 变成瓶颈
- 人怨声载道

改进：
分化风险等级。
- 低风险（修改注释、单元测试）：自动 merge
- 中风险（新增功能）：一个 review
- 高风险（支付、权限）：多个 review + 测试
```

### ❌ 流程太松散

```
"随便让 AI 改，改完就上线"

后果：
- 出问题频繁
- 无法追责
- 团队对 AI 产生不信任

改进：
建立基础的 Harness。
- CI 自动检查不能少
- 高风险改动需要人工 review
- 要有监控告警
```

### ❌ 规格过时不更新

```
现象：
- 规格写了一次就没改过
- 代码实现了但规格没更新
- 新人看规格，但实际代码不一样

后果：
- AI 按过时规格生成代码
- 新人被误导
- 规格信息效力下降

改进：
规格改动和代码改动必须同步。
如果修改了代码，就要更新规格。
```

## 实践建议

### 对个人

1. **写规格时考虑可复用性**。这次规格好，下次团队能复用。
2. **记录你的决策**。为什么这样设计，为什么这个方案被拒了，这都是知识。
3. **参与评审**。别人的规格有问题，提意见；别人的 Skill 有遗漏，补充。

### 对团队

1. **从小做起**。不是一下子建立完整的流程，而是逐步沉淀。
2. **让规格 review 成为文化**。规格评审和代码 review 一样重要。
3. **定期更新知识库**。每个季度清理一遍过时的规格和约束。
4. **培训新人**。新人入职第一件事是学规格库。

### 对管理

1. **给 AI 编程时间投入**。建立规范需要时间，不要急。
2. **鼓励总结和分享**。最佳实践的积累比快速完成更重要。
3. **衡量 AI 的效果**。看指标，看成本，看质量，综合评估。
4. **有问题时不要甩锅给 AI**。要看是规格不清楚还是流程有漏洞。

## 总结

组织流程工程是把 AI 编程从个人工具升级到团队能力的最后一步。

从"这个项目里用 AI 不错"升级到"我们有系统的方法在团队中大规模使用 AI"。

关键是：**把隐性知识显式化，把口头规则写下来，把个人经验变成团队财富**。
