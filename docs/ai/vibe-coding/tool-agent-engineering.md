# 工具与 Agent 工程：可验证的行动闭环

## 定义

工具与 Agent 工程是让 AI 不仅能"建议怎么做"，还能"自己动手做，然后验证做对了"的实践。

从"AI 回答问题"升级到"AI 完成任务"的关键转变。

## 为什么需要工具与 Agent

### 手动执行的问题

```
人（用户）：请帮我优化这个 API 的性能。

AI（提示词工程）：以下是 5 个优化建议...

人（用户）：怎么验证这些建议有效？
AI：你需要 benchmark 一下...

人（用户）手动执行：
1. 改代码
2. 跑性能测试
3. 对比数据
4. 问 AI "这是好还是不好"
5. AI 继续建议...

结果：两个小时过去了，问题还没完全解决。
```

### Agent 工程的优势

```
人：帮我优化这个 API。这是代码、这是现在的性能数据。

Agent（自动化）：
1. 读取代码 ✓
2. 分析性能瓶颈 ✓
3. 提议优化方案 ✓
4. 改代码 ✓
5. 跑性能测试 ✓
6. 对比改前改后 ✓
7. 如果更快了就提交；如果变慢了就回滚 ✓
8. 汇报结果："优化完成，QPS 提升 30%，新增代码 20 行。" ✓

结果：30 分钟完成，人类不需要手动操作。
```

## Agent 的工作循环

```
┌─────────────────────────────────────────┐
│  1. 理解任务（读提示词和上下文）        │
└──────────────┬──────────────────────────┘
               │
┌──────────────▼──────────────────────────┐
│  2. 制定计划（拆成可执行的小步骤）      │
└──────────────┬──────────────────────────┘
               │
┌──────────────▼──────────────────────────┐
│  3. 执行步骤（调用工具：读文件、改代   │
│     码、运行命令、打开浏览器等）        │
└──────────────┬──────────────────────────┘
               │
┌──────────────▼──────────────────────────┐
│  4. 观察结果（看工具的输出和错误）      │
└──────────────┬──────────────────────────┘
               │
┌──────────────▼──────────────────────────┐
│  5. 判断是否完成                        │
│  ├─ 完成了？→ 汇报结果，完成           │
│  ├─ 失败了？→ 分析错误，调整计划       │
│  └─ 继续中？→ 执行下一步               │
└──────────────┬──────────────────────────┘
               │
          ──────────── ◄── 循环
```

## 必要的工具

### 1. 文件系统工具

Agent 需要能：

```
- 读文件（理解现状）
  $ cat src/api/payment.js
  
- 改文件（执行改动）
  $ vi src/api/payment.js
  
- 删文件（清理）
  $ rm src/temp.js
  
- 创建文件夹（组织代码）
  $ mkdir src/utils
```

### 2. 命令行工具

Agent 需要能运行项目的标准命令：

```
- 测试
  $ npm run test
  $ pytest
  
- 类型检查
  $ npm run type-check
  $ mypy src/
  
- 构建
  $ npm run build
  $ cargo build
  
- Lint
  $ npm run lint
  $ black .
  
- 性能测试
  $ npm run benchmark
  $ ab -n 1000 http://localhost:3000
```

### 3. 版本控制工具

Agent 需要能：

```
- 查看改动
  $ git diff
  
- 提交代码
  $ git add .
  $ git commit -m "..."
  
- 创建分支
  $ git checkout -b feature/...
  
- 查看历史
  $ git log
  
- 回滚
  $ git revert HEAD
```

### 4. 浏览器工具

Agent 需要能：

```
- 打开页面
  $ open http://localhost:3000
  
- 填表单
- 点按钮
- 截图
- 读 console 错误
- 检查网络请求
```

### 5. 数据库工具

Agent 需要能：

```
- 查询
  $ psql -c "SELECT * FROM users WHERE id=123"
  
- 插入测试数据
  $ sqlite3 test.db < seed.sql
  
- 查看 schema
  $ psql -c "\\d users"
```

## Agent 的能力等级

### 等级 1：信息采集 Agent

能力：
- 读文件、查看目录
- 运行命令看输出
- 总结和分析

用途：
- "检查一下这个函数的类型错误"
- "看看最近的性能指标"
- "整理一份变更总结"

```
用户：检查这个代码有没有类型问题。
Agent：
1. 读取文件 → 看到代码
2. 运行 `npm run type-check` → 看到 TypeScript 错误
3. 分析 → "第 45 行的参数类型不匹配"
4. 汇报
```

### 等级 2：修复 Agent

能力：
- 等级 1 的全部
- 改文件
- 提交代码

用途：
- "修复这个 bug"
- "重构这个函数"
- "优化这个查询"

```
用户：修复第 45 行的类型错误。
Agent：
1. 读文件 → 看到代码
2. 运行检查 → 确认问题
3. 改文件 → 修复错误
4. 运行测试 → 验证修复
5. 提交 → 完成
```

### 等级 3：特性开发 Agent

能力：
- 等级 2 的全部
- 创建文件和文件夹
- 跑集成测试
- 查看前端效果

用途：
- "添加一个新 API 端点"
- "实现用户登出功能"
- "优化列表加载性能"

```
用户：添加用户导出功能。
Agent：
1. 分析现有代码 → 理解架构
2. 设计方案 → 给出计划
3. 创建新文件 → routes/export.js
4. 写逻辑 → 连接 API
5. 写测试 → 覆盖主要场景
6. 运行所有测试 → 验证不破坏现有功能
7. 本地测试 → 开启开发服务器，测试 UI
8. 提交 → 完成
```

### 等级 4：交付 Agent

能力：
- 等级 3 的全部
- 创建和更新 PR
- CI/CD 流程
- 数据库迁移

用途：
- "完成这个特性并创建 PR 等审核"
- "部署这个变更到测试环境"
- "执行数据库迁移"

## 工具链的规范化

### 1. 项目 README 中定义核心命令

```markdown
## 开发工作流

```bash
# 安装依赖
npm install

# 开发环境
npm run dev  # 启动开发服务器，localhost:3000

# 验证代码
npm run type-check  # TypeScript 类型检查
npm run lint        # 代码风格检查
npm run test        # 单元测试
npm run test:e2e    # 端到端测试

# 构建和部署
npm run build       # 生产构建
npm run start       # 启动生产服务
```

对 Agent 来说，这些就是它可以依赖的"保证"。
```

### 2. 自动化验证（CI/CD）

让这些命令在每次提交时自动运行：

```yaml
# .github/workflows/ci.yml
name: CI

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
      - run: npm install
      - run: npm run type-check
      - run: npm run lint
      - run: npm run test
      - run: npm run build
```

如果任何一步失败，PR 就不能合并。Agent 也要通过同样的检查。

### 3. 快速反馈循环

Agent 需要能快速知道"我的改动对不对"：

```bash
# 秒级反馈
npm run type-check    # TypeScript 错误
npm run lint          # 代码风格
npm run test          # 单元测试失败

# 分钟级反馈
npm run build         # 构建错误
npm run test:e2e      # 集成测试

# 小时级反馈
部署到测试环境       # 真实环境验证
```

## Spec-Driven 的 Agent 工程

规格在 Agent 工程中的角色：

### 1. 规格即测试

好的规格包含可验证的条件，这些条件可以变成测试：

```
规格说：
"评分功能必须在点击后立即更新 UI，不能等待网络请求。"

可以变成测试：
✓ 点击星星后立即看到 UI 改变
✓ 网络失败时 UI 仍然改变（有错误提示）
✓ 网络成功时后端数据也更新了

Agent 运行这些测试验证规格达成。
```

### 2. 规格引导 Agent 的计划

```
规格：
- 支持 Google OAuth（已有后端支持）
- 支持邮箱登录（需要新增邮箱验证逻辑）
- 登录后直接跳转到首页

Agent 制定计划：
1. 读现有的认证代码和后端 API 规格
2. 查看 Google OAuth 集成是否完成（看后端代码和测试）
3. 新增邮箱登录逻辑（API 调用 + UI）
4. 补充邮箱验证逻辑
5. 测试两种登录方式
6. 验证登录后跳转到首页
7. 提交
```

### 3. 规格作为验收标准

```
Agent 改完代码后，自动检查：
☐ 功能满足规格吗？ → 运行相关测试
☐ 性能满足规格吗？ → 运行性能测试
☐ 代码满足规格吗？ → 运行 lint 和类型检查
☐ 没有破坏现有功能？ → 运行全量测试

全部通过才能提交。
```

## Agent 工程的风险控制

### 1. 沙箱和隔离

```
✓ Agent 在独立分支上工作
✓ Agent 不能直接推送到 main
✓ Agent 改动要人工 review 后才能合并
✓ Agent 不能访问生产数据库
```

### 2. 权限最小化

```
✓ Agent 只能读必要的文件
✓ Agent 只能运行白名单内的命令
✓ Agent 不能改配置文件（除非明确允许）
✓ Agent 不能删除重要文件
```

### 3. 每一步都可观测

```
✓ 记录 Agent 的所有操作
✓ 保存改动的 diff
✓ 记录所有命令的输出
✓ 记录测试结果
✓ 如果失败，保留现场方便 debug
```

### 4. 快速回滚

```bash
✓ 如果改动有问题，一键回滚
git revert <commit>

✓ 如果改动有风险，创建新分支重试
git checkout -b retry/...

✓ 重要操作前创建 checkpoint
git tag checkpoint-before-major-change
```

## 工具不能替代的东西

### ❌ 架构决策

Agent 能优化代码，但不能决定"是用微服务还是单体"。

### ❌ 需求理解

Agent 能按规格写代码，但不能理解"用户真实想要什么"。

### ❌ 风险判断

Agent 能测试，但风险是否可接受需要人工评估。

### ❌ 跨系统协调

Agent 能改一个服务，但不能决定整个系统的迁移策略。

## 实践建议

### 对个人

1. **把常用命令写进项目**。`npm run test`, `npm run build` 一定要有。
2. **让每个命令都幂等**。运行两次不能产生不同结果。
3. **提供清晰的验证命令**。告诉 Agent "怎么知道改好了"。
4. **保留检查点**。改复杂功能时分阶段提交。

### 对团队

1. **建立工具链标准**。全 team 用同一套命令。
2. **CI/CD 检查要严格**。有 CI 检查，Agent 才敢自动改。
3. **定期清理工具链**。删掉过时的脚本，更新新的最佳实践。
4. **监控 Agent 操作**。看 Agent 在怎么使用工具，优化工具链。

## 总结

工具与 Agent 工程把"AI 建议"变成"AI 行动"。

从"这是我的想法，请看看"升级到"这是规格，请执行、验证、汇报"。

关键是：**有明确的工具链、有验证命令、有回滚方案**。
