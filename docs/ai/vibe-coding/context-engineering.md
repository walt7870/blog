# 上下文工程：让 AI 看到真实情况

## 定义

上下文工程是精心组织和提供信息，让 AI 准确理解项目现状、技术约束和任务背景的实践。

好的上下文就像一份完整的工作资料包：轻量、相关、易消化。

## 为什么需要上下文工程

### 提示词的局限

即使提示词写得再好，AI 也需要看到真实的代码才能准确动手。

```
提示词说：
"优化 API 的性能，目标 <200ms。"

但 AI 不知道：
- 现在的 API 是什么样的？（可能已经用了缓存）
- 数据库查询在哪里卡顿？（需要看日志和 SQL）
- 有哪些已有的优化？（防止重复优化）
- 哪些改动被禁止？（可能改不了某个库的版本）

只靠提示词，AI 可能生成完全错误的优化方向。
```

### 上下文的作用

给 AI 看以下信息后，它就能做出准确的判断：

1. 现有 API 代码
2. 数据库 schema 和慢查询日志
3. 现有的缓存策略
4. 性能测试报告
5. 框架版本和依赖约束
6. 已尝试过的优化方向（失败或成功）

有了真实背景，AI 的建议才靠谱。

## 好的上下文应该具有的特征

### 1. 高度相关（只放当前任务需要的）

```
❌ 坏的做法：
"这是我们整个项目的源码，请帮我看一下。"
（上传了 10MB 的代码库）

✓ 好的做法：
"这是和支付相关的源码（4 个文件，2000 行）。
还有最近 7 天的支付失败日志和我们使用的 Stripe 文档摘要。"
```

相关性强的上下文让 AI 快速定位问题，而不是淹没在无关信息里。

### 2. 摘要优先（长文件要总结）

```
❌ 上传整个 1000 行的数据库迁移脚本

✓ 提供摘要：
"这是 users 表迁移脚本。
目的：添加 verified_email 字段。
影响：2M+ 行记录，迁移策略是分批处理。
关键限制：不能在生产高峰期运行（下午1-3点）。"

然后再提供关键部分代码片段。
```

### 3. 结构清晰（按逻辑分类，不是按文件）

```
❌ 混乱的上下文：
这是 utils.js、services.js、models.js、config.js 的代码...

✓ 有组织的上下文：
【认证流程】
- 登录 API 端点（services/auth.js 第 45-80 行）
- Token 生成逻辑（utils/jwt.js）
- 权限检查中间件（middleware/auth.js）

【数据库层】
- User 模型（models/user.js）
- 现有的数据库连接（config/db.js）

【依赖和约束】
- 使用 JWT，不能改成 Session
- 密码必须 hash（已用 bcrypt）
```

按业务逻辑分组，AI 更容易理解全景。

### 4. 决策和约束明确化

```
❌ 只有代码，没有背景：
POST /api/users { name, email, password }

✓ 带上约束和决策：
POST /api/users { name, email, password }

为什么这样设计：
- Email 作为唯一标识，支持重置账号
- Password 前端已 hash，后端再 hash 一次（defense in depth）
- 没有返回 plain password（安全考虑）

已有的验证：
- Email 格式校验（使用 validator 库）
- Password 长度 >= 8（需要后端强制）

后续约束：
- 不能改 Email 作为主键的设计（会影响现有数据库）
- 新增字段需要创建迁移脚本（不能直接改 schema）
```

## 上下文的组织方式

### 针对代码改动任务

```
【代码现状】
- 改动涉及的文件列表（带行号）
- 关键函数的目的和署名
- 现有的测试覆盖

【输入上下文】
1. 核心源码（必读）
   - 你要修改的文件（完整）
   - 它依赖的关键模块（摘要）
   
2. 测试例子（看懂行为）
   - 相关的单元测试（完整）
   - 集成测试场景（摘要）
   
3. 业务规则（明白约束）
   - 相关的文档
   - 已知的 bug 和 workaround
   
4. 部署和工具（知道怎么验证）
   - 构建命令
   - 测试命令
   - 部署步骤（如果需要的话）

【背景和决策】
- 为什么代码现在是这样的
- 之前尝试过什么方案，为什么没用
- 有什么技术债或已知问题

【限制】
- 哪些文件不能改
- 哪些依赖版本不能改
- 哪些 API 不能变更（影响其他模块）
```

### 针对架构设计任务

```
【系统现状】
- 系统架构图（或文字描述）
- 主要模块和数据流
- 现在的瓶颈或问题

【输入上下文】
1. 关键模块的代码（摘要）
2. 监控数据和性能指标
3. 用户量和增长预测
4. 已有的扩展方案评估

【约束和选择】
- 技术栈不能改（原因）
- 团队规模和能力
- 成本预算
- 时间紧迫程度

【成功标准】
- 性能目标（QPS、延迟、可用性）
- 可维护性要求
- 风险承受度
```

### 针对 bug 修复任务

```
【Bug 现象】
- 用户可复现的步骤
- 错误日志或 stack trace
- 发生频率和影响范围

【输入上下文】
1. 相关代码（完整）
2. 错误日志（最近 24h，有时间戳）
3. 部署日志（看看最近改了什么）
4. 用户操作的上下文（如果有的话）

【已尝试的修复】
- 之前改过什么，效果怎样
- 为什么那些改动没用

【环境信息】
- 什么版本的代码出现的问题
- 什么环境可以复现（开发、测试、生产）
- 浏览器/系统版本（如果是前端问题）
```

## 常见的上下文错误

### ❌ 上下文太多

```
问题：
"我上传了 50MB 的项目源码，你帮我分析一下。"

后果：
- AI 读取缓慢，响应变慢
- AI 容易抓不住重点
- Token 消耗巨大，成本高

改进：
"这 50MB 代码中，支付相关的部分在 src/payment/ 目录。
我给你提供：
1. 支付模块的核心代码（500 行）
2. 最近改动的 commit（200 行 diff）
3. 支付流程的文档（2 页）
4. 相关的失败日志（100 行）

总共约 5MB，够你分析的。"
```

### ❌ 上下文太少

```
问题：
"优化这个函数的性能。"
（没有提供函数代码）

后果：
AI 只能猜测，建议通常无法实行。

改进：
提供完整的函数代码、调用栈、性能数据。
```

### ❌ 上下文杂乱无章

```
问题：
先给代码、再给日志、再给架构图、再给邮件记录...

后果：
AI 需要反复跳跃来理解，容易造成理解错误。

改进：
按逻辑分类：
【现状】...
【背景】...
【限制】...
```

### ❌ 上下文过时

```
问题：
提供了 3 个月前的代码快照，不知道最近改了什么。

后果：
AI 基于过时信息生成代码，可能和现有代码冲突。

改进：
提供最新代码。如果代码太多，提供最近一周的改动 diff。
```

## 上下文工程的工具

### 1. 从代码库智能提取

**工具**：git、grep、ls-tree

```bash
# 获取最近改动的文件
git log --name-only -n 5

# 找到相关代码
grep -r "function updatePayment" --include="*.js"

# 获取特定文件的历史
git log -p src/payment/handler.js | head -200
```

### 2. 自动生成摘要

**工具**：AI 模型、文档生成工具

```bash
# 用 AI 生成代码摘要
claude "请总结这个函数做了什么。代码：[...]"
```

### 3. 上下文包管理

创建项目根目录下的 `.claude/context/` 文件夹：

```
.claude/
  context/
    payment-module.md       # 支付模块上下文包
    auth-system.md          # 身份验证系统上下文包
    database-schema.md      # 数据库上下文包
    dependencies.md         # 依赖约束
```

每个包包含：
- 相关代码摘要和全文
- 关键决策和约束
- 已知的 bug 或 workaround
- 相关的测试和文档

使用时：提供 context 文件的链接或内容。

### 4. 项目 README 中的上下文清单

```markdown
## 快速 AI 协作

如果让 AI 改这个项目，先给它这些信息：

- **起点**：[ARCHITECTURE.md](./ARCHITECTURE.md) - 架构总览
- **验证**：`npm run test && npm run build`
- **关键约束**：[CONSTRAINTS.md](./CONSTRAINTS.md)
- **已知 bug**：[KNOWN_ISSUES.md](./KNOWN_ISSUES.md)

常见任务的上下文包：
- [支付流程](./docs/context/payment.md)
- [认证系统](./docs/context/auth.md)
- [数据库](./docs/context/database.md)
```

## 上下文的三个层级

### L1：快速修复（小改动）

只需要：现有代码 + 测试 + 相关约束

```
"修复这个 bug：
代码：[...]
现有测试：[...]
约束：不能改 API 署名
"
```

### L2：功能开发（中等改动）

需要：代码 + 测试 + 架构 + 决策背景 + 部署步骤

```
"添加支付功能：
模块结构：[...]
现有支付相关代码：[...]
后端 API 设计（已定好）：[...]
测试要求：[...]
验证命令：npm run test:payment
"
```

### L3：系统重构（大改动）

需要：完整的系统理解、性能数据、成本约束、时间表

```
"重构数据库层：
现有架构和性能指标：[...]
问题分析：[...]
约束条件：[...]
迁移计划和风险：[...]
验收标准和监控：[...]
"
```

## 实践建议

### 对个人

1. **养成收集上下文的习惯**。改代码前 5 分钟内把相关信息整理好。
2. **定期更新项目的上下文资料**。新增大改动时，顺手写进项目 wiki。
3. **给 AI 的上下文，之后要给新员工**。两份资料越接近越好。
4. **上下文是可复用的**。同类型任务直接套用。

### 对团队

1. **建立上下文标准**。全team 用同一套结构。
2. **上下文放在项目 git 里**。和代码版本同步。
3. **定期清理过时的上下文**。去掉已修复的 bug、过时的约束。
4. **让 AI 帮你维护上下文**。"请生成一份这个模块的上下文包"。

## 上下文工程的边界

### 上下文无法解决的问题

- ❌ 提示词本身模糊 → 需要提示词工程
- ❌ 架构决策有问题 → 需要人工设计评审
- ❌ 验证需要真实运行 → 需要工具和 agent
- ❌ 涉及多个系统的集成 → 需要更大的规划

## 总结

好的上下文是**让 AI 快速进入工作状态的垫脚石**。

从"这里有 50GB 的代码，你看着办"到"这是支付模块，5 个文件，2000 行，核心逻辑在这里，约束是 X，测试命令是 Y"，AI 能做的质量差别巨大。
