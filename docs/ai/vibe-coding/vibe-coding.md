# Vibe Coding：原型快速验证

## 定义

Vibe Coding 是快速表达和验证想法的开发方式。用自然语言描述需求，AI 根据描述生成代码或界面原型，人快速看效果、测试基本流程、决定是否继续迭代。

这个阶段的核心是**速度和反馈**，而不是完美和严谨。

## 适用场景

### 非常适合

- **产品想法验证**：新功能是否真的需要，用户是否会用。做一个最小化原型测试。
- **UI/交互探索**：不确定界面应该怎么布局，快速做几个方案让人看。
- **技术方案评估**：某个库或框架是否适合，快速打一个 demo 看效果。
- **临时工具**：一次性脚本、数据处理工具、构建辅助脚本。
- **低风险的原型**：不涉及真实数据、不会被用户直接使用、可以推倒重来。

### 不适合

- 生产环境代码。
- 涉及支付、权限、数据隐私的功能。
- 需要长期维护的核心系统。
- 安全性要求高的模块。

## 工作流

### 1. 清楚地表达想法

```
我想做一个待办事项列表，需要能：
1. 添加新任务（输入框 + 按钮）
2. 标记完成或未完成（复选框）
3. 删除任务（垃圾桶图标）
4. 显示已完成任务数 / 总任务数

界面要简洁，用 React + TailwindCSS。
```

好的表达包含：功能列表、交互方式、技术栈。避免"做一个很牛的 app"这样的模糊描述。

### 2. 看 AI 生成的初版

AI 生成代码后，快速：

- **跑起来**：本地开发服务器启动，看界面。
- **试一遍**：点击按钮、输入数据，走一遍基本流程。
- **找问题**：界面哪里不对劲，功能缺少什么，样式丑吗。

### 3. 快速反馈和迭代

反馈要具体：

```
✓ 好的反馈：
"添加任务后输入框应该自动清空，现在是留着前面的文字"

✗ 模糊反馈：
"感觉有点不对"
```

每一轮反馈后重新生成，看效果。通常 2-3 轮就能出一个可看的原型。

### 4. 决定下一步

- **继续迭代**：原型OK，继续往里加功能。
- **停止**：想法不值得追，推倒重来。
- **进入工程化**：原型验证成功，现在要把它变成可维护的代码。

## Vibe Coding 的常见陷阱

### 1. "感觉对了就交付"

原型完成后，直接用到生产环境，没有补测试、没有处理错误、没有考虑并发。结果出问题的时候，代码一团糟，没人敢改。

**解决**：原型做完决定继续开发时，立即补上[提示词工程](./prompt-engineering.md)和[上下文工程](./context-engineering.md)的规范要求。

### 2. 依赖不明确

生成的代码使用了某个 npm 包，但没记下版本、没查过这个包是否可靠、没看过许可协议。

**解决**：即使是原型，`package.json` 也要清楚。告诉 AI："列出所有依赖和版本号"。

### 3. 没有基本的错误处理

原型在"一切顺利"的情况下能用，但网络请求超时、用户输入奇怪字符、浏览器崩溃时就挂。

**解决**：原型可以简化，但一定要加最基础的 try-catch 和用户提示。

### 4. 代码结构太随意

所有逻辑堆在一个文件，组件混乱，函数命名奇怪。原型阶段懒得整理，后来改都改不了。

**解决**：Vibe Coding 可以快，但文件夹和组件分层不费力。让 AI："按以下文件夹结构组织"。

## Vibe Coding 中的 Spec-Driven 做法

虽然 Vibe 阶段是快速和灵活的，但也应该有最小规格：

### 验证场景清单

写下你想通过这个原型验证的 3-5 个关键场景，原型做完逐一测试：

```
需要验证的场景：
☐ 用户能快速添加任务
☐ 任务列表能显示超过10个任务而不崩溃
☐ 标记完成后任务外观明显改变
☐ 删除后任务真的消失了
☐ 页面刷新后数据还在（如果需要持久化的话）
```

### 最小验收标准

```
原型合格标准：
✓ 上述5个场景都能跑通
✓ UI 在手机和桌面都能看
✓ 没有明显的 console error
✓ 响应时间<100ms（如果有网络请求的话）
```

### 已知限制和风险清单

```
这个原型不处理的：
- 多用户同时编辑
- 数据加密或隐私
- 离线使用
- 无障碍访问 (a11y)

后续如果要到生产，这些都需要补。
```

## 何时从 Vibe 进入工程化

### 原型继续迭代的信号

- 功能已经稳定，不再大改。
- 代码超过 500 行开始有复杂度。
- 有多个人需要理解和修改。
- 要在真实环境测试，涉及真实数据。
- 预计后续要维护超过3个月。

**此时应该**：

1. 补完[提示词规格](./prompt-engineering.md)。
2. 补完[上下文资料](./context-engineering.md)。
3. 补完测试。
4. 接入[工具验证](./tool-agent-engineering.md)。
5. 架构评审。

## 推荐工具和技术栈

### 前端原型

- **UI 框架**：React + TailwindCSS（快速、视觉好）
- **开发服务器**：Vite（热加载快）
- **部署**：Vercel 或 GitHub Pages（秒级）

### 后端原型

- **框架**：Node.js Express 或 Python FastAPI（快速启动）
- **数据库**：SQLite 或 PostgreSQL local（本地测试）
- **工具**：Postman 或 curl（快速测试 API）

### 全栈

- **一体框架**：Next.js、Remix（前后端一起写）
- **BaaS 服务**：Firebase、Supabase（数据库和认证开盒即用）

## 从原型到生产的升级清单

如果原型要变成真实产品，check 这些：

- [ ] 所有函数都有单元测试
- [ ] 关键流程有集成测试
- [ ] 生产构建通过 CI 检查
- [ ] 有错误监控（Sentry、ELK 等）
- [ ] 数据库有备份和恢复流程
- [ ] 权限和身份验证实现了
- [ ] 安全漏洞扫描通过
- [ ] 文档写了（代码注释 + 部署说明）
- [ ] 人工测试了主要用例
- [ ] 有回滚和 hotfix 计划

## 总结

Vibe Coding 是让想法快速变成看得见摸得着的东西的方式。它的力量在于**反馈循环快**，但危险也在这里——快容易滑向"不负责任的快"。

解决方案是：**快不等于乱**。即使在原型阶段，也可以在 5 分钟里定好 3-5 个验证场景、定好技术栈、定好文件结构。这样快的原型也是可管理的原型，后续升级时不会推倒重来。
