# 提示词工程：清晰的任务规格

## 定义

提示词工程是用结构化、精确的语言描述任务，让 AI 理解真实意图、准确输出结果的实践。

这个阶段把含糊的想法变成可验证的规格。

## 为什么需要提示词工程

### Vibe Coding 中的问题

```
模糊的需求：
"给这个页面加个搜索功能"

AI 生成的可能结果 A：
搜索框在顶部，实时搜索，高亮关键词

AI 生成的可能结果 B：
搜索框在右上角，搜索后跳转到结果页，没高亮

结果都"对"，但都不是用户想要的。
```

提示词工程解决的是这个信息差。

## 提示词的结构

### 标准结构

一个好的提示词应该包含以下部分：

#### 1. 任务目标（做什么）

```
目标：为在线学习平台的课程卡片添加评分功能
```

清晰、名词化、不模糊。

#### 2. 范围限制（改哪些、不改哪些）

```
改什么：
- 在课程卡片底部添加 5 星评分组件
- 用户点击星星后立即更新显示
- 保存评分到数据库

不改什么：
- 不修改课程列表的 HTML 结构
- 不改变现有的样式体系（用 BEM 规范）
- 不处理权限校验（假设后端已检查）
- 不涉及评论功能
```

范围明确后，AI 和人对"完成"的理解就一致了。

#### 3. 输入上下文（先读什么）

```
先读以下文件，理解现有代码：
- src/components/CourseCard.tsx（卡片组件）
- src/api/courses.ts（API 调用）
- src/styles/theme.css（样式变量）
- docs/api-spec.md（后端接口规格）
```

告诉 AI 哪些是重要的，哪些可以忽略。

#### 4. 验收标准（怎样算完成）

```
验收标准：
☐ 评分组件在卡片底部正确显示
☐ 点击星星后立即更新本地状态和远程数据库
☐ 新增的代码通过 TypeScript 类型检查
☐ 相关单元测试通过（至少3个场景）
☐ 没有 console warning 和 error

运行命令验证：
npm run type-check  # 类型检查
npm run test -- CourseCard  # 单元测试
npm run build  # 生产构建
```

验收标准要可执行、可验证。

#### 5. 风险和禁区

```
风险和禁区：
⚠ 注意用户可能快速连点星星，需要去抖动
⚠ 后端返回不同格式时要降级处理
❌ 禁止在组件里直接操作 DOM
❌ 禁止使用全局变量存储评分数据
❌ 禁止发送未签名的请求到后端
```

告诉 AI 什么是"可接受但风险"，什么是"绝对不行"。

### 例子：完整提示词

```
=== 任务：添加课程评分功能 ===

【目标】
为在线学习平台的课程卡片添加 5 星评分功能。
用户点击星星评分，立即更新显示并保存到数据库。

【范围】
改什么：
- 在 CourseCard 组件底部添加评分 UI
- 添加点击星星的事件处理
- 调用后端 API 保存评分
- 显示当前用户的评分和平均评分

不改什么：
- 卡片的 HTML 结构
- 现有样式（继续用 TailwindCSS）
- 权限校验逻辑（后端负责）
- 评论/回复功能

【输入上下文】
必读：
- src/components/CourseCard.tsx - 现有卡片组件
- src/api/courses.ts - API 工具
- tailwind.config.js - 样式配置

参考：
- docs/api-spec.md 中的 PUT /courses/{id}/rating 接口规格
- 已有组件 src/components/StarRating.tsx（可复用）

【验收标准】
功能：
☐ 显示 5 个星星，支持点击评分（1-5）
☐ 显示当前用户评分和平均评分
☐ 点击后立即更新 UI
☐ 请求失败时显示错误提示并回滚 UI

代码质量：
☐ 通过 TypeScript 类型检查（npm run type-check）
☐ 通过所有相关单元测试（npm run test -- CourseCard）
☐ 构建成功（npm run build）
☐ 没有控制台错误和警告

性能：
☐ 评分请求<200ms 响应
☐ 不产生多余的渲染

【风险点】
⚠ 用户可能快速点击多个星星 → 需要去抖动
⚠ 网络不稳定 → 需要重试机制
⚠ 并发请求 → 后端应处理并发，前端禁止并发提交
❌ 禁止直接修改 DOM
❌ 禁止使用 localStorage 存储敏感数据
❌ 禁止修改现有 API 调用方式（会影响其他功能）
```

## 常见错误

### ❌ 模糊的提示词

```
"优化这个页面，让它更快更好用。"

问题：
- 什么叫"快"？ 多少毫秒是目标？
- 什么叫"好用"？ 改哪些交互？
- "优化"有无限多种可能性。
```

### ❌ 遗漏重要上下文

```
"添加登录功能。"

问题：
- 现有的身份验证体系是什么？
- 用哪个 oauth 提供商？
- 存在哪些约束？（如：不能改现有数据库结构）
```

### ❌ 验收标准不可验证

```
"代码要足够好。"

问题：
- "足够好"是主观的。
- 没有可执行的验收命令。
```

### ✓ 改进后的版本

```
"改善登录页面的加载时间到<2s，并支持 Google OAuth。

范围：
- 只改 src/pages/login/
- 使用现有的 google-auth 库
- 后端已支持 Google OAuth，直接调用

验收：
☐ 首屏加载<2s（lighthouse）
☐ 支持 Google 登录（能完整走通流程）
☐ 测试通过（npm run test -- login）
☐ 无 TypeScript 错误

禁区：
- 不改现有数据库结构
- 不改权限系统
"
```

## 提示词的三个层级

### L1：快速原型（Vibe 阶段）

```
目标和几个关键点就够了。

例：
"做一个 React 计数器，点 +1 按钮增加，点 -1 按钮减少。
用 TailwindCSS 样式。要看起来简洁。"
```

### L2：功能开发

```
完整的五要素（目标、范围、上下文、验收、风险）。

用于：团队中等规模功能、需要多人 review、会被维护一段时间。
```

### L3：关键路径（支付、权限、数据迁移）

```
五要素 + 详细的数据流、异常处理、审计日志、回滚方案。

用于：任何涉及钱、安全、数据完整性的操作。
```

## 提示词写给谁

### 给 AI（模型）的好处

- AI 明白真实需求，减少胡乱猜测。
- 可以按照约束来生成代码。
- 更容易被 AI 自查（"我生成的满足规格吗"）。

### 给人类 review 者的好处

- Review 时有明确的标准。
- 减少"我觉得这样更好"的无限改进。
- 易于写出可追踪的 review 意见。

### 给后续维护者的好处

- 代码为什么这样写，规格里都有。
- 改代码时知道红线在哪里。

## 提示词工程的工具和实践

### 1. 提示词模板

```
=== 提示词模板 ===
【任务】
【目标】
【范围】- 改什么、不改什么
【输入上下文】
【验收标准】
【风险点】
【备注】
```

保存到项目的 `docs/prompt-template.md`，让 AI 助手和团队一起参考。

### 2. 版本管理

提示词也是代码资产，放进 git：

```
docs/
  prompts/
    feature-auth.md
    feature-payment.md
    refactor-api-layer.md
```

### 3. 提示词库

高频重复的任务写成可复用的提示词模板，比如：

- "新增 API 端点" → `prompt-new-api.md`
- "优化 React 组件" → `prompt-optimize-component.md`
- "写单元测试" → `prompt-write-tests.md`

### 4. Review 提示词

修改后再提示词，是优化的关键：

```
第一版提示词：
"添加评论功能"
↓ AI 生成，看起来一般般
↓ 补充提示词

第二版提示词：
"添加评论功能。注意：
- 评论要支持 markdown 格式
- 要防止 XSS 攻击
- 需要分页（每页20条）
- 要有评论审核流程"
↓ AI 重新生成，好多了
```

## 提示词工程的限制

**提示词不能解决的**：

- ❌ 缺失的背景知识（如：现有系统架构）→ 需要上下文工程
- ❌ 代码质量验证 → 需要工具和测试
- ❌ 架构决策 → 需要人工 review
- ❌ 涉及真实用户的验证 → 需要部署和真实测试

**提示词加上下文**：

如果提示词里的任务无法仅通过文字说清楚，需要 AI 看代码、日志、API 响应，那就进入[上下文工程](./context-engineering.md)。

## 实践建议

### 对个人

1. **写提示词前先想清楚**。30 秒的思考省 30 分钟的返工。
2. **第一版提示词会不完美**。写出来，让 AI 生成，再改。
3. **保存好的提示词**。做过一次的功能，提示词可以复用。
4. **每次改进代码时，顺手更新提示词**。让后来人受益。

### 对团队

1. **建立提示词规范**。让所有人用同一套五要素结构。
2. **定期更新提示词库**。把新学到的约束加进去。
3. **提示词作为需求的真实来源**。优于模糊的 PRD。
4. **提示词 review**。新增大功能前，规格先过 review。

## 总结

好的提示词是**任务和规格的精确翻译**。从"差不多这样吧"升级到"满足这10个条件就算完成"。

提示词工程不是为了让流程变复杂，而是为了**减少误解、加快迭代、降低返工**。
