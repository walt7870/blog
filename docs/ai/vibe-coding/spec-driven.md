# Spec-Driven 开发：从感觉到规格的转变

## 概述

Spec-Driven（规格驱动）是让 AI 编程从"感觉对了"升级到"满足规格"的开发方法论。

它回答了一个关键问题：**代码什么时候算"完成"——是凭直觉，还是凭规格？**

## 传统开发 vs Spec-Driven

### 传统方式（感觉驱动）

```
产品经理：我想要一个搜索功能，让用户快速找到内容。

开发者：好的，我来做。
（几天后...）

展示给产品：做好了！

产品经理看了看：嗯...感觉不太对。
页面有点慢，结果的排序方式也不是我想的。

开发者：那怎么改？

产品经理：让用户能按日期排序，搜索结果要在 500ms 内返回。

开发者再改...（返工开始）
```

问题：
- ❌ "搜索功能"太模糊，谁都可以有不同理解
- ❌ 改动太后期，已经写了代码
- ❌ 无法验证"做完了没有"
- ❌ 返工成本高

### Spec-Driven 方式

```
产品经理编写规格：
"搜索功能规格

目标：用户能通过关键词快速找到内容

验收标准：
☐ 搜索结果在 500ms 内返回（包括网络延迟）
☐ 支持按相关性和日期排序（默认按相关性）
☐ 显示相关性分数（0-100）
☐ 搜索框支持自动完成（从现有数据生成）
☐ 没有搜索结果时有友好提示

限制：
- 不修改现有数据库 schema（只用现有字段）
- 使用现有的搜索引擎（Elasticsearch）
- 保持现有的 UI 整体风格"

产品和开发一起 review 规格，确保理解一致。

开发者按规格编码。

完成后验证：
- [ ] 搜索结果在 500ms 内返回？ ✓
- [ ] 支持两种排序？ ✓
- [ ] 显示相关性分数？ ✓
- [ ] 自动完成？ ✓
- [ ] 错误提示？ ✓

规格全满足，任务完成。没有返工。
```

好处：
- ✓ 规格清晰，减少误解
- ✓ 提前沟通，避免后期返工
- ✓ 验收标准明确，减少争议
- ✓ 可以自动化验证

## Spec-Driven 的核心原则

### 1. 规格优先于实现

```
传统顺序：
想法 → 代码 → 验证 → 改改改

Spec-Driven 顺序：
想法 → 规格 → 代码 → 验证（通常一次通过）
```

规格不是代码的附属物，而是代码的输入。

### 2. 规格是可执行的

好的规格不是模糊的文字说明，而是可以变成测试的约束：

```
✗ 模糊规格：
"搜索要快"

✓ 可执行规格：
"搜索结果在 500ms 内返回（测试 1000 条记录，3 个并发查询）"

这可以变成自动化性能测试：
benchmark('search', () => {
  const start = Date.now();
  const results = search('query', 1000);
  expect(Date.now() - start).toBeLessThan(500);
});
```

### 3. 规格是真实来源

```
当代码和规格冲突时，规格赢。

例：
规格：搜索结果必须按相关性排序
代码：搜索结果按日期排序

→ 代码是 bug，必须修正。

规格永不改，除非产品确认需求改了。
```

### 4. 规格包含失败情况

好的规格不仅说"成功时怎样"，也说"失败时怎样"：

```
规格的完整性：

成功路径：
☐ 用户输入有效关键词 → 返回搜索结果

边界情况：
☐ 用户输入空字符串 → 显示所有内容
☐ 用户输入特殊符号 → 转义或返回错误提示
☐ 搜索结果超过 10000 条 → 分页显示

失败路径：
☐ 搜索引擎不可用 → 显示"暂时无法搜索"
☐ 网络超时 → 自动重试，超过 3 次后显示错误
☐ 用户权限不足 → 只显示有权限的结果
```

## 在 AI 编程中应用 Spec-Driven

### 1. 给 AI 的提示词就是规格

```
❌ 模糊的提示词：
"给这个页面加个搜索功能"

✓ 规格形式的提示词：
"【任务】添加搜索功能

【验收标准】
☐ 搜索结果在 500ms 内返回
☐ 支持按相关性和日期排序
☐ 显示相关性分数
☐ 搜索结果为空时提示"无结果"

【限制】
- 只修改 src/components/Search.tsx
- 不改 API 接口（后端已有搜索端点）
- 样式遵循 TailwindCSS 规范

【验证命令】
npm run test -- search.test.ts
npm run lint
npm run build"
```

AI 按规格工作，完成后自动验证规格。

### 2. 规格即 AI 的 review 标准

不再问"代码看起来怎样"，而问"规格满足了吗"：

```
人工 review 清单：

规格验收：
☐ 所有 9 个验收标准都满足了吗？
  （检查 test output）

代码质量：
☐ 通过类型检查（npm run type-check）
☐ 通过 Lint（npm run lint）
☐ 测试覆盖率 >= 80%

风险评估：
☐ 改动是否超出规格范围？
☐ 有没有隐藏的假设？
☐ 依赖改动是否安全？

→ 如果全部通过，直接 merge。
```

### 3. 规格帮助 AI 自查

AI 完成改动后，可以自动检查：

```
AI 自查流程：

1. 我的改动满足了规格中的所有验收标准吗？
   → 运行相关测试
   
2. 代码通过了基本检查吗？
   → npm run type-check && npm run lint && npm run build
   
3. 我是否不小心改了禁区？
   → 检查改动的文件列表
   
4. 我有没有删除或弱化现有功能？
   → 运行所有测试，检查失败数

5. 我是否引入了新的依赖？
   → 检查 package.json 的改动

总结：
✓ 规格 9/9 满足
✓ 代码检查全部通过
✓ 所有测试通过
✓ 无新依赖，无超范围改动

结论：✓ 准备 review
```

## Spec-Driven 的三个层级

### L1：快速原型（简化规格）

```
适合：个人探索、低风险功能、临时脚本

规格可以简化，但必须有：

【目标】
【核心验收标准】（3-5 个）
【禁区】
【验证命令】

例：
【目标】实现计数器功能
【验收标准】
☐ 点 +1 按钮数字增加
☐ 点 -1 按钮数字减少
☐ 刷新页面数字保存（localStorage）
【禁区】不改现有的页面结构
【验证】npm run dev（手动测试）
```

### L2：功能开发

```
适合：新功能、涉及多个模块、需要维护 6 个月以上

完整的规格五要素：

【目标】清晰明确
【范围】详细说明改什么、不改什么
【输入上下文】相关的代码、文档、已有实现
【验收标准】详细的成功/失败/边界情况
【风险点】已知的难点和约束
```

### L3：关键路径（支付、权限、架构）

```
适合：支付、权限、数据迁移、架构重构

完整规格 + 附加内容：

五要素 + 以下部分：
【数据流图】系统如何协作
【失败恢复】出错时怎么回滚
【审计日志】怎么追踪
【性能目标】具体的数字
【灾难恢复】最坏情况的应急方案
【涉及人员】谁需要 sign-off
```

## Spec-Driven vs 敏捷开发

### 关系

```
Spec-Driven 和敏捷并不矛盾。
实际上 Spec-Driven 是更好的敏捷实践。

敏捷的问题：
"快速迭代"经常变成"快速返工"
因为需求不清楚，改完一轮再改一轮。

Spec-Driven 敏捷：
在小 sprint 内（比如 1 周），
提前和产品沟通，定好这个 sprint 的规格。
然后开发和 AI 都按规格工作。

结果：快速迭代变成快速交付。
```

## Spec-Driven 的常见误区

### ❌ 误区 1：规格就是完整需求文档

```
错误理解：
"要写 50 页的需求文档才能开始编码"

正确理解：
规格是精炼的、可验证的标准。
可以短到半页纸（对于小功能）。

例：
完整需求文档：10 页，讲背景、市场、用户故事...
规格：1 页，讲清楚验收标准和限制
```

### ❌ 误区 2：规格写完就不改了

```
错误理解：
规格一旦定下来就永不改变

正确理解：
在开发过程中可能发现规格有遗漏或不切实际。
这时应该更新规格，但要：
1. 必须 review（不能一个人随意改）
2. 要通知所有相关人
3. 改动必须记录（为什么改的）

规格是活文档，但改动要规范。
```

### ❌ 误区 3：所有东西都要规格

```
错误理解：
"每个代码改动都要规格"

正确理解：
按重要性分化：
- 关键改动：需要详细规格
- 常规改动：需要轻量规格
- 紧急修复：只需要 bug 描述和简单验证

别把小事也搞得很复杂。
```

### ✓ 做对的方式

```
规格的粒度要匹配工作量：

1 小时以内的任务 → 轻量规格（几行）
1-3 天的任务 → 标准规格（半页到一页）
1 周以上的任务 → 详细规格（1-3 页）
系统级改动 → 完整规格 + 架构设计
```

## 实践步骤

### 步骤 1：定义验收标准

在开始编码前，列出 3-10 个清晰的验收标准：

```
✓ 用户能通过关键词搜索
✓ 搜索结果在 500ms 内返回
✓ 结果可以按相关性或日期排序
✓ 相关性分数以百分比显示（0-100）
✓ 结果为空时显示友好提示
✓ 搜索框支持自动完成
✓ 搜索历史保留最近 10 条（localStorage）
✓ 在低网速下也能用（降级方案）
```

### 步骤 2：写出对应的测试

把每个验收标准变成可验证的测试：

```
✓ → test('user can search by keyword', () => {...})
✓ → test('search completes within 500ms', () => {...})
✓ → test('can sort by relevance or date', () => {...})
... 以此类推
```

### 步骤 3：编码

代码要满足所有测试。

### 步骤 4：验证

在提交前检查：

```
npm run test -- search.test.ts
# 所有 8 个测试都通过？

npm run type-check && npm run lint && npm run build
# 类型、风格、构建都通过？

# 手动测试关键场景
npm run dev
# 搜索功能看起来对吗？
# 在手机和桌面都能用吗？
```

### 步骤 5：Review

Reviewer 检查：
- 规格的 8 个验收标准都满足了吗（看测试）
- 代码质量过关吗（看 CI）
- 有没有超出规格范围（看 diff）

## Spec-Driven 带来的好处

### 给开发者

- 明确知道什么时候"完成"
- 减少返工
- Review 时间短（标准明确）
- 可以放心重构（规格保证正确性）

### 给产品

- 需求更清晰，减少误解
- 更早发现问题（规格 review 时）
- 上线质量更高

### 给 AI

- 有明确的目标，不用猜
- 可以自检（规格是自测标准）
- 提升成功率

### 给团队

- 知识显式化（规格是文档）
- 复用便捷（好的规格可以用作模板）
- 争议少（有规格就没有"我觉得"）

## 总结

Spec-Driven 是说：**把"感觉对了"升级到"规格满足"。**

这不是为了增加流程，而是为了**减少返工、加快交付、提高质量**。

特别是在 AI 编程的时代，好的规格就像好的提示词一样重要。AI 改代码的质量 80% 取决于规格的清晰程度。
