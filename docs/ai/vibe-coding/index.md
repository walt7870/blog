# 从 Vibe Coding 到工程化 AI 编程

Vibe Coding 指用自然语言描述想法，让 AI 快速生成软件原型，再根据运行效果继续调整。它降低了开始做软件的门槛，但如果只停留在”感觉对了就接受”，项目很快会遇到维护、测试、架构和责任边界问题。

更准确的理解是：Vibe Coding 是入口体验，不是完整工程方法。长期项目需要把提示词、上下文、工具、验证和团队流程逐步补齐。

## Spec-Driven 的核心思想

Spec-Driven（规格驱动）是避免”感觉对了”陷阱的关键方法论。核心思想是：

- **从规格而非感觉出发**：任何功能都应该先有清晰的规格说明，包含行为、边界、输出格式、验收标准。
- **规格作为验收标准**：代码是否完成，不由 review 者的直觉判断，而由是否满足规格来判定。
- **可验证 vs 模糊**：”差不多是这样”是 vibe 阶段，”满足以下10个条件”是规格阶段。
- **上下文固化**：规格、代码、测试、部署说明放在一起，减少信息丢失。

在 AI 编程中，spec-driven 意味着：
1. 给 AI 明确的规格而不是模糊的想法。
2. 用规格而不是演示效果来验证 AI 的输出。
3. 针对不同阶段的规格做差异化要求：原型阶段规格可以简化，但必须有；生产代码的规格必须精确。

这样做的好处：减少误解、让 review 有标准、AI 输出更稳定、后续维护有依据。

## 几个阶段

### 1. Vibe Coding（原型快速验证）

这一阶段重在快速表达想法。用户描述页面、功能或流程，模型产出初版，用户看效果后继续调整。

**适合**：

- 验证产品想法。
- 做低风险原型。
- 探索 UI 或交互方向。
- 临时脚本和小工具。

**风险**：

- 代码结构可能散。
- 边界用例没有覆盖。
- 依赖和安全配置可能随意。
- 人不理解生成物，后续难维护。

**Spec-Driven 的做法**：即使在原型阶段也应该列出需要验证的3-5个核心场景。

→ [详细说明](./vibe-coding.md)

### 2. 提示词工程（清晰的任务规格）

提示词工程解决”任务怎么说清楚”。好的任务说明至少包含目标、范围、限制、输出形式和验收标准。这是 spec-driven 的关键实践。

**比较稳定的规格表达方式**：

- 改什么。
- 不改什么。
- 先读哪些上下文。
- 完成后怎样验证。
- 最终说明哪些风险。

**关键原则**：提示词能减少误解，但它不能弥补缺失的项目资料。好的提示词是对规格的精确翻译。

→ [详细说明](./prompt-engineering.md)

### 3. 上下文工程（为规格收集充分证据）

上下文工程解决”让模型看到什么”。在代码任务中，模型需要看到相关源码、测试、配置、错误日志、项目规范和运行命令。

**好的上下文像一份资料包**：

- 只放当前任务相关内容。
- 长材料先摘要。
- 决策和约束写进文档。
- 不同任务隔离上下文。
- 工具结果要短而准。

**Spec-Driven 的视角**：上下文是为了让 AI 理解规格的背景。上下文太少会让模型猜；上下文太乱会让模型抓错重点。

→ [详细说明](./context-engineering.md)

### 4. 工具与 Agent 工程（让规格变成可验证的闭环）

当 AI 能读文件、改文件、运行命令、打开浏览器、查日志时，它从”回答建议”进入”执行任务”。这时规格的验证从人工审阅变成可自动化验证。

**关键变化是闭环**：

1. 读取项目。
2. 制定计划。
3. 修改文件。
4. 运行验证（规格的可执行部分）。
5. 根据错误继续修复。
6. 汇报变更和风险。

**Spec-Driven 的要求**：规格必须包含可执行的验证命令（测试、lint、构建等）。没有验证命令的 Agent 只能声称完成；有验证闭环的 Agent 才能证明完成。

→ [详细说明](./tool-agent-engineering.md)

### 5. Harness 工程（规格的强制执行）

Harness 是围绕 Agent 的工程化外壳，包括规则、工具、沙箱、测试、日志、回滚和评审。它的目标是让规格得到强制执行，而不仅仅是建议。

**常见组成**：

- 仓库说明：技术栈、目录、命令、禁区。
- 可执行检查：测试、lint、类型检查、构建（规格的门槛）。
- 隔离环境：分支、worktree、容器、沙箱。
- 可观测性：日志、截图、trace、接口响应（验证规格的工具）。
- 评审机制：Agent 自查、人类 review、CI 阻断。
- 恢复机制：checkpoint、回滚、任务拆分。

**设计理念**：Harness 像轨道和护栏。没有它，AI 速度越快，风险也越快放大。

→ [详细说明](./harness-engineering.md)

### 6. 组织流程工程（让规格成为系统常识）

当团队开始大量使用 AI 编程后，真正稀缺的不是生成代码，而是判断标准。团队需要把架构原则、质量红线、发布流程、安全规则和用户体验标准写进系统。

**有效做法**：

- 把重复 review 意见变成规则、测试或 Skill。
- 把需求拆成规格、验收和回归用例。
- 把高风险动作设置人工确认点。
- 定期清理过时文档，避免 Agent 学到旧规则。

**Spec-Driven 的组织实践**：建立规格库、验收标准库、风险清单库，作为团队的共识。

→ [详细说明](./organizational-engineering.md)

## Spec-Driven 开发方法论

从”感觉对了”到”规格满足”是 AI 编程必须要走的路。

→ [深入理解 Spec-Driven](./spec-driven.md)

## 个人使用建议

- 原型阶段可以快，但长期项目要尽早补测试和目录约束。
- 大任务拆成小任务，每一步都能看 diff 和验证。
- 让 AI 先读相关文件，再动手改。
- 把常用命令写进项目文档。
- 两次以上重复出错，就把规则写成 Skill 或测试。
- 对鉴权、支付、数据迁移、生产配置保持人工 review。

## 团队使用建议

- 先改善仓库可理解性，再追求更多自动化。
- 让测试更容易运行，错误信息更容易定位。
- 让文档成为系统事实来源，而不是聊天记录。
- 给 Agent 最小权限，不给不必要的生产写权限。
- 让 CI 检查代码、文档、生成物和架构边界。
- 人类 review 聚焦需求、风险、架构和用户体验。

## 总结

Vibe Coding 是让想法快速变成看得见摸得着的东西的方式。它的力量在于**反馈循环快**，但危险也在这里——快容易滑向”不负责任的快”。

工程化 AI 编程的六个阶段回答了这个问题：

1. **快速表达**（Vibe Coding）→ 证明想法可行
2. **清晰说明**（提示词工程）→ 减少误解
3. **充分准备**（上下文工程）→ 给 AI 正确信息
4. **可验证行动**（工具 Agent 工程）→ 从建议到执行
5. **安全框架**（Harness 工程）→ 让行动可控
6. **系统复用**（组织流程工程）→ 让知识和规范扩展到团队

越是长期项目，越不能只依赖”感觉对了”。Spec-Driven 的核心是把验收标准从”模糊的主观印象”升级到”清晰的可验证规格”。

**结果**：快速不再等于乱，迭代不再等于返工。
