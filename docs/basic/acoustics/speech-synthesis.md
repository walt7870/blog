# 语音合成

语音合成 TTS 把文本转换成可播放的语音。它用于语音助手回复、导航播报、客服外呼、无障碍阅读、听书和虚拟人。

## 基本链路

```text
文本 -> 文本规范化 -> 分词/韵律预测 -> 声学模型 -> 声码器 -> 音频后处理 -> 播放
```

| 模块 | 作用 |
| --- | --- |
| 文本规范化 | 把数字、日期、单位、符号转成可读形式 |
| 分词与多音字 | 判断词边界和读音 |
| 韵律预测 | 决定停顿、重音、语速和语调 |
| 声学模型 | 生成梅尔谱等声学特征 |
| 声码器 | 把声学特征还原为波形 |
| 后处理 | 响度、限幅、采样率、格式转换 |

## 文本规范化

TTS 最常见错误不是声音不好听，而是读错文本。

| 原文 | 可能读法 | 需要规则 |
| --- | --- | --- |
| 2026-05-17 | 二零二六年五月十七日 | 日期 |
| 3.14 | 三点一四 | 小数 |
| 100km/h | 每小时一百公里 | 单位 |
| RMB 128.5 | 一百二十八点五元 | 金额 |
| 010-88888888 | 零一零，八个八 | 电话 |

文本规范化要按业务控制。财务、医疗、导航、客服对数字和单位的读法差异很大。

## 多音字和韵律

中文 TTS 需要处理多音字和断句。例如“重庆”和“重复”的“重”读音不同，“银行行长”里两个“行”的读音也不同。

韵律决定一句话是否自然：

- 停顿位置是否符合语义。
- 重点词是否突出。
- 疑问句、感叹句语调是否正确。
- 长句是否切分成可听的短语。
- 列表、金额、地址是否读得清楚。

## 声学模型与声码器

现代 TTS 常使用神经网络生成语音。工程上常见两段式结构：声学模型生成声学特征，声码器生成波形。

| 组件 | 关注点 |
| --- | --- |
| 声学模型 | 音色、韵律、稳定性、可控性 |
| 声码器 | 音质、实时率、算力消耗 |
| 说话人编码 | 多音色或声音克隆 |
| 风格控制 | 情绪、语速、音高、强弱 |

实时交互更关注首包延迟和流式合成，长文本播报更关注稳定性、自然度和长句一致性。

## 流式合成

流式 TTS 边生成边播放，适合语音助手和实时对话。

需要处理：

- 文本分片：不能等完整回答才开始合成。
- 语义边界：分片不能破坏句子读法。
- 首包延迟：用户感知明显。
- 播放中断：用户打断时要能停止。
- 缓冲策略：网络抖动不能导致播放断续。

如果文本来自大模型流式输出，TTS 不应每来几个字就合成一次。更稳妥的做法是按标点、短语或语义边界切片。

## 音色与声音克隆

音色用于塑造品牌和角色。声音克隆可以用少量样本模仿目标说话人，但涉及授权、隐私和滥用风险。

工程约束：

- 明确音色使用授权。
- 对敏感场景增加水印、审核或权限控制。
- 避免让合成声音冒充真实个人身份。
- 对未成年人声音和客服场景设置更严格规则。

## 播放侧处理

TTS 输出还要进入实际播放链路。好听的离线音频不一定在设备上好听。

播放侧需要关注：

- 采样率是否和系统播放链路一致。
- 响度是否和提示音、音乐、通话音量协调。
- 是否出现削顶、底噪、爆音。
- 小喇叭是否能承载低频。
- 播放时是否影响 AEC 和唤醒。

小体积设备通常需要对低频做限制，避免扬声器失真。提示音和 TTS 的响度要统一，否则用户会觉得系统忽大忽小。

## 质量评估

| 指标 | 说明 |
| --- | --- |
| MOS | 主观听感评分 |
| 首包延迟 | 从提交文本到第一段音频可播放的时间 |
| 实时率 RTF | 合成耗时 / 音频时长，低于 1 才能实时 |
| 读错率 | 多音字、数字、单位、英文缩写读错比例 |
| 中断恢复 | 播放中断、继续、重播是否稳定 |

TTS 评估要同时看可懂度、自然度、稳定性和业务正确性。客服和导航场景里，读准比声音华丽更重要。

## 常见问题

| 现象 | 可能原因 | 处理 |
| --- | --- | --- |
| 数字读错 | 文本规范化规则不足 | 增加领域规则 |
| 长句怪异 | 分句和韵律预测差 | 做文本切分和标点修正 |
| 首句响应慢 | 非流式合成、模型冷启动 | 流式合成、预热、缓存 |
| 播放破音 | 输出响度过高、功放削顶 | 限幅、统一响度 |
| 小喇叭发闷 | 频响受限、低频过多 | EQ 和设备侧适配 |

## 总结

TTS 的工程质量来自文本处理、模型能力和播放链路共同作用。业务系统应把文本规范化、流式切片、音量策略、打断机制和播放回声一起设计，而不是只接入一个合成接口。
