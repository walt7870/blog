# 语音唤醒

语音唤醒用于判断用户是否说出了指定唤醒词。它让设备在大部分时间保持低功耗监听，只在用户明确呼叫后进入识别和交互。

## 唤醒在链路中的位置

![唤醒、识别与合成的关系](/acoustics/speech-ai-modules.svg)

唤醒模块通常运行在端侧，因为它需要常开、低延迟、可离线。云端唤醒会带来隐私、延迟和网络依赖问题。

## 核心指标

| 指标 | 含义 | 风险 |
| --- | --- | --- |
| 唤醒率 | 正确说出唤醒词后被唤醒的比例 | 太低会让用户反复喊 |
| 误唤醒率 FAR | 没说唤醒词却被唤醒的概率 | 太高会打断用户、增加隐私风险 |
| 漏唤醒率 FRR | 说了唤醒词但没唤醒的概率 | 影响可用性 |
| 首响延迟 | 从说完唤醒词到设备反馈的时间 | 太长会显得迟钝 |
| 功耗 | 常开监听消耗 | 影响电池设备续航 |

唤醒阈值是典型取舍：阈值降低，唤醒率提高但误唤醒变多；阈值提高，误唤醒减少但漏唤醒增加。

可以把唤醒模型理解成一个“门卫”。它不是完整理解用户在说什么，而是持续判断当前声音是否足够像指定唤醒词。阈值就是开门标准：标准太松，路过的相似声音也会进来；标准太严，真正用户也可能被挡在门外。

## 唤醒词设计

好的唤醒词要有区分度、易读、自然、少歧义。

设计建议：

- 不要太短，两个字容易和日常词混淆。
- 不要太长，用户成本高，模型检测窗口也变长。
- 避免高频生活词、品牌名歧义和常见口头禅。
- 尽量包含清晰辅音和节奏变化。
- 考虑普通话、方言、儿童声音和口音。

唤醒词不是只看模型。产品命名、市场传播和用户习惯都会影响实际唤醒数据。

## 模型与流程

常见唤醒流程可以拆成三层：先由 VAD 判断“有没有人在说话”，再由 KWS 判断“说得像不像唤醒词”，最后由确认逻辑判断“是否连续、稳定、可信”。这比单帧判断更稳，因为真实唤醒词有一段持续的发音轮廓，而很多误触发只是在某个瞬间相似。

为避免唤醒后吞掉用户后续命令，系统通常会保留一段音频缓存。例如用户说“你好设备，打开空调”，唤醒词后的“打开空调”不能因为 ASR 启动慢而丢失。

## 多阶段唤醒

低功耗设备常用多阶段架构：

| 阶段 | 位置 | 作用 |
| --- | --- | --- |
| 一级唤醒 | DSP、低功耗 MCU | 常开监听，模型小，功耗低 |
| 二级确认 | 主芯片或更大模型 | 复核唤醒，降低误唤醒 |
| 声纹确认 | 可选 | 判断是否为授权用户 |

多阶段能降低功耗和误唤醒，但会增加延迟和实现复杂度。

## 声纹与说话人识别

声纹不是唤醒词本身，而是判断“谁在说”。它常用于个性化、儿童保护、权限控制。

适合场景：

- 车载里区分主驾、副驾和后排。
- 智能家居里识别家庭成员。
- 设备执行敏感指令前做轻量确认。

不适合把声纹当作唯一安全凭证。录音重放、环境噪声、感冒、年龄变化都会影响声纹稳定性。高风险操作仍应结合账号、设备、密码或二次确认。

## 测试集设计

唤醒测试不能只用安静房间里的标准读音。

测试集应覆盖：

- 距离：近场、1 米、3 米、5 米。
- 角度：正面、侧面、背面。
- 噪声：电视声、音乐、车噪、厨房、办公室、人群。
- 播放：设备自播音乐或 TTS 时唤醒。
- 人群：成人、儿童、男女声、方言口音。
- 负样本：相似词、包含部分音节的句子、日常聊天。

误唤醒评估需要长时间背景音，不能只看几分钟演示。

## 常见问题

| 现象 | 可能原因 | 处理 |
| --- | --- | --- |
| 远处唤不醒 | SNR 低、阵列方向错、阈值过高 | 查分路录音、BF 输出和阈值 |
| 播放音乐时唤不醒 | AEC 不足、播放参考缺失 | 查 ERLE 和参考信号 |
| 经常误唤醒 | 唤醒词太像常用词、阈值低、负样本不足 | 补负样本、二级确认、调阈值 |
| 唤醒后第一字丢失 | ASR 启动慢、缓存不足、VAD 截断 | 增加前后缓存 |
| 儿童唤醒差 | 训练数据不足、频段差异 | 补儿童数据或适配模型 |

## 总结

语音唤醒的核心是“可用性、误唤醒、功耗、延迟”的平衡。上线前应把唤醒率和误唤醒率放到真实环境长时间测试中验证，并保留唤醒前后的音频片段用于复盘。
