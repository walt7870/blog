# 业务架构（Business Architecture）

## 定义与目标

业务架构定义了企业的业务战略、组织能力、业务流程与治理结构，旨在确保业务目标与企业架构的对齐，为信息系统与技术架构提供清晰的业务蓝图。

业务架构不是简单画流程图。它首先回答“企业到底需要哪些能力来实现战略”，再回答“这些能力如何通过组织、流程、规则和信息协同起来”。后续数据架构、应用架构和技术架构都应从业务能力出发，而不是从已有系统清单出发。

## 主要内容

- 业务愿景与目标
- 业务能力（Business Capability）地图
- 价值流（Value Stream）与客户旅程
- 业务流程与服务（Business Process/Service）
- 组织结构与角色职责
- 业务信息与规则
- 合规与治理要求

## 关键工件

- 业务愿景文档与目标树
- 能力地图（一级/二级/三级能力）
- 价值流与价值实现链
- 端到端业务流程模型（BPMN）
- 组织与角色矩阵（RACI）
- 业务信息模型与词汇表

## 方法与步骤（对应ADM Phase B）

1. 明确业务驱动与范围
2. 建立业务原则与约束
3. 现状业务架构基线（Baseline）
4. 目标业务架构蓝图（Target）
5. 业务差距分析（Gap Analysis）
6. 路线图与机会识别
7. 风险评估与依赖关系
8. 与数据/应用/技术架构对齐

## 模型与视图

- 能力视图：展示企业的能力构成与成熟度
- 价值流视图：从客户需求到价值交付的端到端链路
- 组织视图：组织结构与协作关系
- 流程视图：关键流程、子流程与活动
- 信息视图：核心业务对象与关系

## 例子：零售全渠道业务架构

一家零售企业从线下门店扩展到线上商城时，业务架构要先把能力拆清楚，而不是直接讨论要不要建订单中心。

| 业务能力 | 子能力 | 架构关注点 |
| --- | --- | --- |
| 会员运营 | 会员注册、等级、权益、触达、复购 | 会员是否有统一身份，权益规则是否能跨渠道复用 |
| 商品管理 | 商品资料、价格、上下架、类目 | 商品主数据由谁维护，门店和线上是否共用商品口径 |
| 交易订单 | 下单、支付、取消、退款、售后 | 订单状态是否统一，跨渠道订单是否可以统一查询 |
| 库存履约 | 可售库存、锁定库存、仓库发货、门店自提 | 库存扣减规则是否一致，履约路径是否可优化 |
| 客户服务 | 工单、投诉、退换货、回访 | 客服能否看到完整会员、订单、物流和售后记录 |

能力地图的价值在于帮助组织识别重复建设和缺失能力。例如如果门店、商城、小程序各自维护会员等级，就会导致权益不一致；如果每个渠道都有自己的订单规则，后续统一售后和数据分析会很困难。

## 度量与治理

- 关键结果（OKR/KPI）与能力成熟度
- 合规与风险控制矩阵
- 架构原则与标准的遵循度
- 业务连续性与韧性指标

## 最佳实践

- 以业务能力为中台：能力与服务化设计
- 价值导向：以价值流优化跨部门协同
- 轻量化与可视化：便于沟通与决策
- 渐进式演进：小步快跑持续优化

## 交付与产物

- 业务架构蓝图与能力地图
- 业务流程体系文档
- 治理与合规模型
- 业务与IT对齐的需求清单

## 关联关系

- 指导数据域与主数据的识别
- 驱动应用组合与服务地图设计
- 约束技术平台的非功能性需求

## 总结

业务架构是企业架构的起点与核心。它把战略翻译成能力、价值流、组织职责和业务规则，再把这些内容传递给数据、应用和技术架构。没有清晰的业务架构，后续系统拆分和技术选型很容易变成局部优化。
