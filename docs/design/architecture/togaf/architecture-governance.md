# 架构治理（Architecture Governance）

## 定义与目标

架构治理确保企业架构实践的一致性、合规性与有效性，通过组织、流程、标准与度量，实现架构的可持续改进与价值交付。

治理不是为了让架构师多盖章，而是为了让目标架构在项目执行中不被一点点冲散。企业架构通常跨部门、跨系统、跨年度推进，如果没有治理机制，目标蓝图很容易在需求变更、交付压力和局部优化中失真。

## 组织与职责

- 架构委员会（ARB）：战略与决策
- 领域架构师：业务/数据/应用/技术
- 解决方案架构师：项目层面的架构落地
- 数据与安全负责人：治理与合规

## 治理机制

- 原则与标准管理（制定、发布、更新）
- 评审机制：方案评审、设计评审、上线评审
- 例外与偏差管理（豁免与补偿措施）
- 一致性检查与合规审计

## 流程与工件

- 架构申报与评审流程
- 决策记录（ADR）与知识沉淀
- 标准目录与参考架构库
- 度量指标与看板

## 迁移路线与实施治理

TOGAF 强调目标架构不能一步到位，通常需要工作包、过渡架构和迁移路线。

| 治理对象 | 说明 | 例子 |
| --- | --- | --- |
| 工作包 | 可以独立推进的一组架构改造任务 | 统一会员中心、订单状态标准化、库存事件改造 |
| 过渡架构 | 从现状到目标之间的阶段性架构 | 先让新订单走订单中心，旧订单仍保留在原系统查询 |
| 架构评审 | 检查项目方案是否符合目标架构 | 新营销活动是否复用会员中心，是否绕过统一权益规则 |
| 例外管理 | 对暂时无法遵循标准的情况登记和补偿 | 某旧系统暂时不能接入统一认证，需要明确下线时间 |
| 变更反馈 | 把实施中的新约束反馈到架构模型 | 发现库存同步延迟超过预期，调整事件补偿和监控方案 |

## 例子：防止目标架构被绕开

假设企业目标是建设统一会员中心，但某个业务团队为了赶活动，想在自己的系统里新增一套会员等级字段。架构治理要判断这是不是短期例外，还是会破坏长期目标。

合理的治理处理方式是：

1. 先确认业务诉求是否真实紧急。
2. 判断统一会员中心是否已有能力或可快速扩展。
3. 如果必须临时绕开，记录例外原因、影响范围、补偿措施和退出时间。
4. 在上线前要求同步数据口径、权限和审计方案。
5. 活动后把临时能力收敛回统一会员中心。

这样治理不是阻塞业务，而是防止短期方案变成长期债务。

## 度量与改进

- 合规度、复用率、技术债与缺陷密度
- 价值交付周期与满意度
- 架构变更响应速度

## 与组织实践的关系

- 与PMO/产品/安全/运维协同
- 与敏捷与DevOps融合：从评审到嵌入流水线

## 最佳实践

- 原则少而精、标准可执行
- 评审轻量化与早期介入
- 知识管理与共享平台
- 持续度量与回顾

## 总结

有效的架构治理在“控制”与“赋能”之间取得平衡。它既要约束风险，也要帮助项目更快复用能力、遵循标准、减少重复建设，并把实施反馈持续沉淀回企业架构资产。
